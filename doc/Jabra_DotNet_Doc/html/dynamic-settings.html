Dynamic Setting API vs. universal API calls

The Jabra SDK v2 provides a long list of universal API calls, however it may not be possible to know if a device supports a specific API call until it has been queried.

To aid UI developers, the Jabra SDK v2 also introduces a concept of ‘Dynamic Settings’. These are device-specific settings which are guaranteed to be accessible and relevant for the connected device. Rather than needing to know API function names at compile-time, Dynamic Settings provide a uniform interface which can be processed at run-time. This means that a UI showing the list of Dynamic Settings will not need to be updated in case new functionality is added, for example after a firmware update, as the list of settings are specific for a given device variant/firmware combination.

Connectivity requirements

Dynamic Settings are retrieved from a cloud backend which means that it requires an online connection in order to use this API. As of this writing, the backend is queried for updates once when a device is attached to the SDK and then again once per hour while the SDK process is active. Prior to downloading any files, the version number of the online files are compared to the local cached versions in order to keep network traffic to a minimum.

Besides the list of settings, the backend will also deliver model device photos in different sizes which can be loaded in a UI for device identification.

The cached files including the device photos are stored in these locations:

Windows: appdata%\JabraSDK\Resources\<PID>_<variant>_<firmware>

Linux/macOS: /tmp/Resources/<PID>_<variant>_<firmware>

Deleting the Resources folder will cause it to be re-created on next SDK init.

If the SDK cannot reach the backend and the local cache is also empty, the normal API functions are still usable, however the returned list of Dynamic Settings will be empty and the feature inactive. If for some reason, the SDK should never attempt to access the backend, this can be requested as part of the configParams parameter when calling InitializeV2().

How to interact with Dynamic Settings

The first step is to get a list of dynamic settings from a device. This is done using this method:

List<ISetting> ReadSettingsFromDevice(ushort deviceId, string guid, ref ErrorStatus error)

Each setting has a unique GUID and if guid != null, it will return a single ISetting object (if the GUID could be found). If guid == null, it will return a List of all available device settings.

Most variables are read-only and meant for UI consumption to define labels and UI element types. The only variable that can be set is CurrentValue if applicable to a given setting (some settings like those that use ControlType.Label are read-only).

Since some settings changes may trigger a device reboot (indicated by IsDeviceRestartRequired=true), the client should update CurrentValue for all elements in the List<ISetting> before finally calling:

In a UI, this flow corresponds to the user toggling settings on/off, editing text fields etc. before finally pressing the Apply button. This will write all changes and reboot the device if necessary. Dynamic Settings are generally persistent and stored in the device. They will only be reset if a factory reset is requested or in case of a firmware upgrade.

The List<ISetting> passed to UpdateSettingsToDevice() does not need to be the full List retrieved from ReadSettingsFromDevice(). It is also possible to pass fewer items or just a single ISetting that needs changing. The SDK will iterate through all elements and apply the changes to only those that had changed from the last communication.

If, for some reason, UpdateSettingsToDevice returned an error code, it usually means that one or more settings could not be updated. This could happen if the device was in use. To get a List of names for those settings which failed to change, this function can be used:

List<IFailedSetting> GetFailedSettingNames(ushort deviceId)

Note that it is not possible to change the DataType or the List inside SettingValues so it is only ControlType.TextBox that allows the user to type in their own value.

The order of the elements in List<ISetting> is stable and logically ordered. Each ISetting belongs to a group which in a UI can be shown as separate tabs. All settings that belongs to the same group will be successive.

These are the text fields contained in an ISetting:

Text element Always defined Description
GroupName yes The headline or title of the group that this setting belongs to.
GroupHelpText no Extended group description. Can be used as a subtitle or hover text of the group.
SettingName yes Title of the setting.
HelpText no Extended description. Can be used as a subtitle or hover text.

While the backend is designed to provide multiple locales, it is currently only possible to choose English.

Input validation

In order for the UI to validate the user input for a setting using ControlType.TextBox before attempting to write to the device, ValidationRules can contain some validation rules (if IsValidationSupported==true).

This is the interface:

public interface IValidations
{
int MinLength { get; }
int MaxLength { get; }
string RegexExp { get; }
string ErrorMessage { get; }
}

For an input string to be acceptable, its Length should be MinLength <= InputStr.Length <= MaxLength and it should have a match with the regular expression found in RegexExp. If not, ErrorMessage can be shown to the user to explain how to enter an acceptable input string.

Settings that can enable/disable other settings

If IsDepedentSettingSupported==true, the device contains some settings which should be able to hide other settings. One example could be to hide IP address and Subnet mask input fields when the Ethernet interface is disabled.

The information to implement this is contained in the SettingValues object. The relevant interfaces are defined like this:

Additional dynamic data

Related to the Dynamic Settings feature, the cloud backend also provides some dynamic data about a connected device. This includes model device photos and a supported feature list.

Accessing device model pictures

To load device photos in the UI, using these methods on the device object will return a string containing the full path to either a full resolution photo or a thumbnail picture, respectively:

string GetDeviceImagePath();
string GetDeviceImageThumbnailPath();

The file format will be PNG and it is then up to the client to load the image from disk.
